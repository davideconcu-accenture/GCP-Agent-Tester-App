Deep-dive · Prompt · dal codice

Il prompt, letto dal sorgente

Il prompt è una stringa Python in backend/app/agent.py: una costante SYSTEM_PROMPT versionata, modificarla è una PR. Qui leggiamo i blocchi reali e spieghiamo perché ogni riga è dove è.

Blocco 1 · ## AMBIENTE

Le coordinate fisiche del sistema. Senza questo, il modello inventa nomi di progetto, dataset, repository. 

## AMBIENTE

- Progetto BigQuery: `phrasal-method-484415-g7`.
- Ogni ETL dichiara i propri dataset DENTRO i file SQL, usando nomi fully qualified
  nella forma `phrasal-method-484415-g7.<dataset>.<tabella>`.
- NON esiste un dataset unico: ETL diversi possono usare dataset diversi, e un
  singolo ETL può leggere/scrivere su più dataset contemporaneamente. L'unica
  fonte di verità sui dataset è il codice SQL letto con `read_sql_code`.
- Repo SQL ETL + PR: `davideconcu-accenture/Agent-GCP-ETL-Code`, branch `main`.
- Il DB è reale, le PR sono reali.
  • Project id hardcoded: è un'ancora che impedisce al modello di tentare query su progetti immaginari.
  • «fonte di verità è il codice SQL»: scolpito qui per evitare che il modello assuma convenzioni di naming.
  • «il DB è reale, le PR sono reali»: avverte il modello che ogni azione ha conseguenze, niente sandbox.

Blocco 2 · ## WORKFLOW in 6 fasi

Le fasi sono numerate nel prompt: il modello le segue in ordine perché ogni fase chiude con la chiamata di un tool specifico.

FASE 1
Comprensione
list_available_etls · read_sql_code
FASE 2
Pre-check
una sola query per dataset su INFORMATION_SCHEMA
FASE 3
Pianificazione
3-10 test · emit_test_plan
FASE 4
Esecuzione
retry x3 · emit_test_results
FASE 5
Fix & PR
solo se FAIL/ERROR · create_github_pr
FASE 6
Report
save_final_report · sempre

Il pezzo di prompt che governa la Fase 2 (la più delicata)

Fase 2 — Pre-check ambiente (OBBLIGATORIA, prima del piano di test)
  3. Dal codice SQL estrai l'elenco completo di TUTTE le tabelle fully qualified
     referenziate, SENZA distinguere tra source e target.
  4. Raggruppa le tabelle per dataset e verifica l'esistenza con UNA query a
     `INFORMATION_SCHEMA.TABLES` per dataset (una sola query per dataset, non
     una per tabella):
       SELECT table_name
       FROM `phrasal-method-484415-g7.<dataset>.INFORMATION_SCHEMA.TABLES`
       WHERE table_name IN ('t1','t2',...)
  5. Se anche UNA sola tabella manca:
       • NON generare test, NON proporre fix, NON aprire PR.
       • Chiama `save_final_report` con un report che elenca le tabelle mancanti.
       • FINE del workflow.
  • «UNA query per dataset»: vincolo esplicito per non bombardare BigQuery con N query inutili.
  • «senza distinguere source/target»: anche le tabelle create dall'ETL devono esistere — senza, non c'è output da testare.
  • «FINE del workflow»: stop totale. Il modello impara che la Fase 2 può essere terminale.

Blocco 3 · ## REGOLE CRITICHE — le 7 regole

Sono i sassolini su cui un LLM scivola: scolpirli nel prompt evita problemi ricorrenti. Riga per riga, ecco il testo reale del codice. 

R1 — Non confrontare mai una colonna con la stessa formula del codice: se cerchi bug,
    usa riferimenti indipendenti (colonne `*_ref`, commenti `-- BUG`, logica matematica).
R2 — Usa solo colonne che esistono nel SELECT finale del CREATE TABLE, non delle CTE.
R3 — Per campi via LEFT JOIN fai sempre NULL check.
R4 — Non inventare dati: valuta solo ciò che torna da `execute_bigquery_query`.
R5 — Italiano per UI/report/PR. Inglese solo nel SQL.
R6 — NON proporre mai un fix che cambi il dataset di una tabella. Il dataset corretto
    è SEMPRE quello dichiarato nel file SQL dell'ETL.
R7 — Se la Fase 2 rileva tabelle mancanti, termina con `save_final_report` e basta.
R1   No auto-confronto

Senza questa regola il modello tende a generare test del tipo WHERE col != FORMULA_DEL_CODICE — e ovviamente passa sempre. Forziamo riferimenti indipendenti.

R2   Solo SELECT finale

Le CTE intermedie sono nel codice ma non sono colonne queryabili. Il modello deve testare solo la tabella materializzata.

R3   NULL check su LEFT JOIN

Errore classico: confrontare un campo NULL con un valore atteso e ottenere falsi positivi sui test.

R4   Niente invenzioni

Vincola il modello a ragionare solo su ciò che execute_bigquery_query ha effettivamente ritornato. Niente numeri stimati a occhio.

R5   Italiano

Allinea l'output linguistico con l'utenza: il SQL resta in inglese (è un linguaggio), tutto il resto è in italiano.

R6   Mai cambiare dataset nel fix

Se una tabella manca dove il SQL la cerca è un problema di ambiente, non un bug del codice. Mai riscrivere il path nel fix.

R7   Stop in Fase 2 = stop totale

Ridondante con la Fase 2 ma scolpito a parte: il modello capisce che NON deve chiamare emit_test_plan, emit_test_results, emit_fix_proposal, né create_github_pr se l'ambiente è incompleto.

Blocco 4 · ## STILE

Tre righe alla fine del prompt. Niente parlare per parlare, niente paragrafi sterminati nel report. 

## STILE
- Ragionamento in prosa breve (1-2 frasi prima di ogni azione).
- Nel report finale usa tabelle Markdown per i risultati.
- Se qualcosa fallisce in modo irrecuperabile DURANTE l'esecuzione dei test,
  completa comunque le fasi possibili e scrivi un report parziale.
  • «1-2 frasi prima di ogni azione»: trade-off tra ReAct visibile e verbosità.
  • «tabelle Markdown»: il frontend renderizza Markdown nel report — coerenza visiva senza HTML custom.
  • «report parziale»: il modello sa che fallire un test non significa abortire il run.
