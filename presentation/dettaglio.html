04 · Dettaglio

L'agente in dettaglio

Un singolo agente IA. Tre ingredienti. Un loop di ragionamento. Vediamo, con parole semplici, cos'è davvero e come funziona.

parte 1

Da cosa è composto l'agente

L'agente è l'integrazione di tre componenti distinti, ciascuno sostituibile in modo indipendente: un modello di linguaggio, un set di istruzioni operative e una libreria di funzioni Python.

modello LLM

Modello Gemini

Large Language Model multimodale di Google, ospitato su Vertex AI. A ogni iterazione interpreta la richiesta, valuta il contesto accumulato e decide se invocare un tool o produrre la risposta finale. Espone la capability di function calling, base operativa del loop ReAct.

  • Default: gemini-2.5-pro
  • Alternativa: gemini-2.5-flash
  • Selezionabile dall'utente al momento della richiesta
istruzioni

System prompt

Testo in italiano che vincola il comportamento dell'agente al dominio specifico. Definisce le sei fasi sequenziali del workflow — dalla lettura del codice alla creazione della Pull Request — e sette regole non negoziabili (es. niente DDL su BigQuery, validazione obbligatoria della fix sui dati reali).

  • Vive nel codice del backend
  • Modificabile senza riaddestrare il modello
  • Versionato come il resto del codice
tool

Otto funzioni Python

Funzioni Python registrate nell'ADK come FunctionTool. Le loro signature tipizzate vengono convertite in JSON Schema e passate al modello, che le invoca quando ha bisogno di leggere o agire sul mondo esterno — BigQuery, GitHub, bus eventi del backend.

  • Lettura SQL da GitHub
  • Esecuzione query BigQuery
  • Apertura Pull Request
  • + 5 funzioni di workflow (piano, risultati, fix, report)
parte 2

Come ragiona — il loop ReAct

L'agente non risponde in un colpo solo. Pensa a voce alta, sceglie uno strumento, osserva il risultato, e ripete. Esattamente come uno sviluppatore davanti a un bug.

Esempio concreto · "verifica i saldi correnti"

PENSA "per partire devo prima leggere il SQL dell'ETL"
AGISCE read_sql_code("saldi_correnti")
OSSERVA riceve il testo del file SQL
PENSA "il SQL fa join tra movimenti e conti — verifico che esistano"
AGISCE execute_bigquery_query("SELECT ... INFORMATION_SCHEMA.TABLES")
OSSERVA tabelle presenti
PENSA "ora pianifico i test mirati ai totali"
… e così via, fino al report finale o alla Pull Request.
