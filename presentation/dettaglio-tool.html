Deep-dive · Tool

Gli 8 tool

Sono normali funzioni Python con type hint e docstring. ADK le wrappa in FunctionTool e le espone al modello come capacità.

Come funziona la registrazione di un tool ADK

from google.adk.tools import FunctionTool

def execute_bigquery_query(query: str, purpose: str) -> dict:
    """Esegue una query SELECT su BigQuery. ..."""
    ...

tool = FunctionTool(func=execute_bigquery_query)
# ADK genera lo schema dai type hint + docstring,
# lo passa al modello e gli permette di chiamarlo.
Famiglia 1 · GitHub (lettura codice)

Esplorazione del repo

list_available_etls() → { etls: [...], repo, branch }

Elenca le cartelle al primo livello del repo SQL su GitHub (un ETL = una cartella). Usato in Fase 1 quando l'utente non ha specificato l'ETL.

read_sql_code(etl_name) → { etl, files: [{path, content}, ...] }

Scende ricorsivamente nella cartella dell'ETL e ritorna tutti i file .sql con il loro contenuto. È il punto da cui l'agente estrae le tabelle fully-qualified.

Famiglia 2 · BigQuery

L'unico tool che parla col DB

execute_bigquery_query(query, purpose) → { columns, rows, row_count }

Esegue una query read-only (solo SELECT/WITH). Limita a 200 righe in fetch e ne mostra al modello al massimo 50. Il parametro purpose è una breve descrizione del test che si sta eseguendo, ed è loggato nella timeline.

Stesso tool, tre usi: pre-check tabelle, esecuzione test, validazione fix.

Famiglia 3 · Comunicazione & azione

Cinque tool

emit_test_plan(tests, rationale)

Pubblica il piano dei test al frontend. Ogni test ha id, name, category, priority, description, query, pass_condition. Salvato come artefatto su Firestore.

emit_test_results(results)

Pubblica gli esiti. Ogni risultato ha id, name, status (PASS | FAIL | ERROR | IGNORED), evidence, row_count.

emit_fix_proposal(root_cause, file_path, original_sql, fixed_sql, fixes_tests, validation_passed, validation_notes)

Pubblica la proposta di fix. Include la causa radice, il SQL originale e quello corretto, gli id dei test che il fix risolve, e — punto chiave — un flag validation_passed che è true solo se l'agente ha già provato la formula corretta su BigQuery.

create_github_pr(branch_name, file_path, new_content, pr_title, pr_body, commit_message)

L'unico tool che esce dal sistema: crea un branch da main, committa il file SQL corretto, apre la Pull Request. Una sola transazione logica. Ritorna url, number, branch (o error).

save_final_report(markdown)

Salva il report finale Markdown del run. Ultimo step di ogni Fase 6, sempre eseguito.

Cosa accade a ogni chiamata di tool

  1. L'ADK Runner emette un evento function_call → il backend lo inoltra come tool_call sul bus SSE.
  2. Il tool esegue, scrive eventi (tool_start, tool_end) e — se è un emit_* o save_* — un artefatto strutturato su Firestore.
  3. L'ADK Runner emette un evento function_response → il backend lo inoltra come tool_response.
  4. Il modello vede il risultato e decide il prossimo passo.
