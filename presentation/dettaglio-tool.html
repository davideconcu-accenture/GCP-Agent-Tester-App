Deep-dive · Tool · dal codice

Gli 8 tool, dal sorgente

Il file è backend/app/tools.py. Otto funzioni Python, nessuna classe, nessun decoratore custom: sono i type hint e la docstring a generare lo schema che ADK passa al modello.

Setup minimale: client lazy, run id contestuale

Tutti i tool condividono tre helper. Una sola istanza per processo, run id recuperato da un ContextVar — non è un parametro del modello. 

from github import Github, GithubException
from google.cloud import bigquery

from app.config import get_settings
from app.store  import CURRENT_RUN_ID, append_event, save_artifact

# Client lazy (una istanza per processo)
_bq: bigquery.Client | None = None
_gh: Github | None = None

def _bq_client() -> bigquery.Client:
    global _bq
    if _bq is None:
        _bq = bigquery.Client(project=get_settings().gcp_project_id)
    return _bq

def _run_id() -> str:
    return CURRENT_RUN_ID.get()    # ContextVar settato dal backend prima del run
  • Lazy init: i client BigQuery / GitHub si creano alla prima chiamata, non all'import — cold start più rapido.
  • CURRENT_RUN_ID è un ContextVar: i tool sanno a quale run appartengono senza che il modello debba passare un id.
  • Niente mock layer: ogni chiamata è reale. Il commento in cima al file lo scolpisce.
Famiglia 1 · GitHub (lettura codice)

Esplorazione del repo

list_available_etls

def list_available_etls() -> dict[str, Any]:
    """
    Elenca le cartelle ETL disponibili nel repository SQL su GitHub.

    Returns:
        Dict con `etls`: lista di nomi di cartelle (es. "etl-banca-italiana").
    """
    repo = _gh_repo()
    s = get_settings()
    contents = repo.get_contents("", ref=s.github_branch)
    etls = [c.name for c in contents
            if c.type == "dir" and not c.name.startswith(".")]
    append_event(_run_id(), "tool_end", {"name": "list_available_etls", "count": len(etls)})
    return {"etls": etls, "repo": s.github_repo, "branch": s.github_branch}

Convenzione semplice: una cartella al primo livello = un ETL. Filtriamo le directory dotfile (es. .github) per rumore zero. Il modello chiama questo tool solo se non sa già quale ETL toccare.

read_sql_code

def read_sql_code(etl_name: str) -> dict[str, Any]:
    """
    Legge tutti i file .sql dentro la cartella di un ETL sul repo GitHub.
    """
    repo = _gh_repo()
    files: list[dict[str, str]] = []

    def _walk(path: str) -> None:
        for c in repo.get_contents(path, ref=s.github_branch):
            if c.type == "dir":
                _walk(c.path)                          # ricorsione
            elif c.name.endswith(".sql"):
                files.append({"path": c.path,
                              "content": c.decoded_content.decode("utf-8")})

    _walk(etl_name)
    return {"etl": etl_name, "files": files}

Discesa ricorsiva nella cartella ETL: l'agente ottiene tutti i .sql in un colpo solo. Da qui estrae le tabelle fully-qualified per la Fase 2.

Famiglia 2 · BigQuery (l'unico tool col DB)

execute_bigquery_query

def execute_bigquery_query(query: str, purpose: str) -> dict[str, Any]:
    """
    Esegue una query BigQuery read-only. Usala per: verifica esistenza tabelle
    (INFORMATION_SCHEMA.TABLES), regole business, null check, duplicati,
    integrità referenziale, correttezza dei join.
    """
    append_event(_run_id(), "tool_start", {"name": "execute_bigquery_query", "purpose": purpose})

    # Guardrail: solo read-only
    stripped = query.strip().upper()
    if not (stripped.startswith("SELECT") or stripped.startswith("WITH")):
        return {"error": "Solo query SELECT/WITH sono ammesse."}

    try:
        job = _bq_client().query(query)
        results = list(job.result(max_results=200))    # fetch limit
    except Exception as e:
        return {"error": str(e)}

    columns = list(results[0].keys())
    rows = [[_to_jsonable(r[c]) for c in columns] for r in results[:50]]   # max 50 al modello
    return {"columns": columns, "rows": rows,
            "row_count": len(results), "truncated": len(results) > 50}
  • Guardrail in 3 righe: strip().upper() + check su SELECT / WITH. Niente DDL, DML, MERGE.
  • Doppio limit: 200 righe in fetch, 50 mostrate al modello + flag truncated. Protegge il context window.
  • purpose obbligatorio: il modello deve dichiarare cosa sta testando — finisce nei log e nella timeline UI.
  • Errori restituiti come {"error": ...}: il modello li legge e può ritentare (max 3 volte da prompt).
Famiglia 3 · Comunicazione & azione

Gli emitter e l'azione finale

Stessa forma per tutti: argomenti tipizzati → save_artifact su Firestore → il backend lo replica via SSE al frontend.

emit_test_plan

def emit_test_plan(tests: list[dict[str, Any]], rationale: str) -> dict[str, str]:
    """
    Pubblica il piano di test al frontend. Chiamala UNA VOLTA dopo aver pianificato
    i test ma prima di eseguirli.

    Args:
        tests: Lista di test. Ogni test ha: id, name, category, priority,
               description, query, pass_condition (es. "zero_rows").
        rationale: Riga di motivazione sul perché hai scelto questi test.
    """
    save_artifact(_run_id(), "test_plan", {"tests": tests, "rationale": rationale})
    return {"status": "ok"}

La docstring elenca esplicitamente i campi attesi nei dict di tests: ADK la passa al modello, che genera JSON conforme al primo colpo.

create_github_pr — l'unico tool che esce dal sistema

def create_github_pr(
    branch_name: str, file_path: str, new_content: str,
    pr_title: str, pr_body: str, commit_message: str,
) -> dict[str, Any]:
    repo = _gh_repo()
    base = repo.get_branch(s.github_branch)

    # 1) crea branch da main (idempotente: 422 = esiste già, ok)
    try:
        repo.create_git_ref(ref=f"refs/heads/{branch_name}", sha=base.commit.sha)
    except GithubException as e:
        if e.status != 422: raise

    # 2) commit del file (update se esiste, create altrimenti)
    try:
        existing = repo.get_contents(file_path, ref=branch_name)
        repo.update_file(file_path, commit_message, new_content, existing.sha, branch=branch_name)
    except GithubException:
        repo.create_file(file_path, commit_message, new_content, branch=branch_name)

    # 3) apri PR verso main
    pr = repo.create_pull(title=pr_title, body=pr_body, head=branch_name, base=s.github_branch)

    result = {"url": pr.html_url, "number": pr.number, "branch": branch_name}
    save_artifact(_run_id(), "pr_opened", result)
    return result
  • Tre chiamate API (branch, file, PR) in una sola transazione logica: dal punto di vista del modello è un solo tool.
  • HTTP 422 = ok: vuol dire che il branch esiste già — tipicamente perché è un retry. Idempotenza voluta.
  • Niente auto-merge: la PR è aperta, il merge è umano. Stop-the-line.

def save_final_report(markdown: str) -> dict[str, str]:
    """Salva il report finale Markdown del run. Ultimo step prima del done."""
    save_artifact(_run_id(), "final_report", {"markdown": markdown, "ts": time.time()})

Tre righe di codice. Sembra banale, ma è il contratto di chiusura: il prompt impone di chiamarlo SEMPRE, anche quando il run è abortito in Fase 2.

Cosa accade a ogni chiamata di tool

  1. L'ADK Runner emette un evento function_call → il backend lo inoltra come tool_call sul bus SSE.
  2. Il tool esegue, scrive eventi (tool_start, tool_end) e — se è un emit_* o save_* — un artefatto strutturato su Firestore.
  3. L'ADK Runner emette un evento function_response → il backend lo inoltra come tool_response.
  4. Il modello vede il risultato e decide il prossimo passo.
