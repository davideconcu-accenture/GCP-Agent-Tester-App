Deep-dive · BigQuery

Dati & query

Il warehouse contiene tabelle sorgente, di staging e finali popolate dagli ETL fittizi. L'agente le interroga sempre in sola lettura.

Setup

  • Progetto GCP: phrasal-method-484415-g7, region us-west1.
  • Dataset multipli: ogni ETL può leggere/scrivere su dataset diversi. La fonte di verità è il SQL, non una convenzione.
  • Tabelle fully-qualified nel codice: project.dataset.tabella. L'agente non inventa nomi, li estrae dal SQL.
  • Client: google-cloud-bigquery con autenticazione ADC tramite service account su Cloud Run.

Pre-check di esistenza

Prima di qualsiasi test, l'agente verifica che tutte le tabelle citate dal SQL esistano. Una sola query per dataset, contro INFORMATION_SCHEMA.TABLES. 

SELECT table_name
FROM `phrasal-method-484415-g7.banca_raw.INFORMATION_SCHEMA.TABLES`
WHERE table_name IN ('clienti','movimenti','saldi')

Se anche una sola tabella manca, l'agente chiude il run con un report di ambiente incompleto: non genera test, non propone fix, non apre PR.

Le query di test

Categorie

  • business_rule — regole funzionali sul dato.
  • data_integrity — duplicati, NULL, referential.
  • join_correctness — verifica dei JOIN.
  • brb_compliance — regole specifiche del cliente.

Convenzione

Predilezione per pass_condition: zero_rows. Una query che torna 0 righe quando va tutto bene è la più robusta e leggibile da implementare. 

-- saldo arrotondato male?
SELECT id_conto, saldo, saldo_ref
FROM `...saldi`
WHERE ABS(saldo - saldo_ref) > 0.01

Guardrail tecnici

  • Solo SELECT / WITH: il tool rifiuta tutto il resto, anche se l'LLM lo prova.
  • Max 200 righe tornate, max 50 mostrate al modello (controllo costi e contesto).
  • Retry intelligenti: se la query fallisce per colonna inesistente, l'agente la riscrive fino a 3 volte. Oltre, marca il test IGNORED.
  • Validazione del fix sempre via SELECT: la formula corretta viene simulata inline per dimostrare che il bug sparisce.
