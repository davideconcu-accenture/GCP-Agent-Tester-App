Deep-dive · GitHub

Repo & Pull Request

Repository SQL versionato, accesso via PyGithub, token su Secret Manager, apertura PR in tre step.

Setup

  • Repo: davideconcu-accenture/Agent-GCP-ETL-Code · branch base main.
  • Una cartella per ETL al primo livello (es. etl-banca-italiana/), file .sql dentro.
  • Tabelle fully-qualified nel SQL: niente convenzioni implicite.
  • Client: PyGithub, una sola istanza per processo (lazy-init).

Token & sicurezza

Secret Manager

In produzione il token GitHub è memorizzato in Google Secret Manager come secret etl-agent-github-token e iniettato nel container come variabile d'ambiente al boot di Cloud Run.

In locale, da file .env (mai committato).

Niente auto-merge

L'agente apre la PR ma non la mergea. Il merge è sempre umano: un revisore valida la causa proposta, l'evidenza dai test, e il SQL corretto.

È il guardrail più importante: l'agente propone, le persone decidono.

Apertura PR · in 3 step dentro un solo tool

Cosa contiene il body della PR

  • Titolo: convenzionale, es. fix(etl-saldi-correnti): corretto ROUND nei totali.
  • Root cause in 2-5 righe in italiano.
  • Test che fallivano e che ora dovrebbero passare.
  • Validazione: la query diagnostica che ha simulato il fix e i suoi risultati.
  • Diff impliito: GitHub mostra il diff, l'agente non lo duplica nel body.

Le anomalie inserite negli ETL

Nel repo SQL abbiamo inserito di proposito sei bug realistici, due per ETL. Sono i bug che l'agente deve scoprire, validare e chiudere con una PR.

ETL 1   Cliente 360

Anomalia 1A — stg_cliente_movimenti

etl_bank_data.sql:40
  • Bug: rimosso il filtro AND m.stato = 'COMPLETATO' dalla CTE mov_cli.
  • Effetto: entrate, uscite e contatori includono anche transazioni ANNULLATO e RIFIUTATO → totali gonfiati, stipendi e movimenti sovrastimati (~10% dei record dato che lo stato è 90% COMPLETATO).
  • Fix: reintrodurre AND m.stato = 'COMPLETATO' nella WHERE.

Anomalia 1B — stg_cliente_conti

etl_bank_data.sql:26
  • Bug: SUM(c.saldo) non filtra per stato = 'ATTIVO'.
  • Effetto: saldo_totale (e quindi value_tier in dm_cliente_360) include i saldi dei conti CHIUSO e BLOCCATO → patrimonio sovrastimato, clienti classificati in tier più alti del dovuto.
  • Fix: SUM(IF(c.stato = 'ATTIVO', c.saldo, 0)), oppure aggiungere WHERE c.stato = 'ATTIVO' gestendo coerentemente il COUNT.

ETL 2   Pacchetto Performance

Anomalia 2A — stg_pacchetto_clienti

etl2_pacchetto_performance.sql:20
  • Bug: COUNTIF(stato <> 'CHIUSO') include anche i clienti in stato SOSPESO tra gli "attivi".
  • Effetto: num_clienti_attivi sovrastimato → performance_tier troppo ottimistico.
  • Fix: COUNTIF(stato = 'ATTIVO').

Anomalia 2B — dm_pacchetto_performance

etl2_pacchetto_performance.sql:76-77
  • Bug: revenue_mensile_stimata e revenue_annua_stimata moltiplicano per num_clienti_totali (che include i CHIUSO), non per num_clienti_attivi.
  • Effetto: revenue ricorrente gonfiata — i clienti churned non pagano più il canone.
  • Fix: usare c.num_clienti_attivi * p.canone_mensile.

ETL 3   Movimenti Mensili

Anomalia 3A — dm_movimenti_mensili_segmento

etl3_movimenti_mensili.sql:42
  • Bug: EXTRACT(MONTH FROM data_operazione) estrae solo il mese, senza l'anno.
  • Effetto: gennaio 2024 + gennaio 2025 vengono collassati nella stessa riga → serie temporale inutilizzabile per trend pluriennali. Classico bug da time-series.
  • Fix: usare DATE_TRUNC(data_operazione, MONTH) AS mese (restituisce DATE, es. 2025-01-01).

Anomalia 3B — dm_movimenti_mensili_segmento

etl3_movimenti_mensili.sql:54
  • Bug: importo_medio è calcolato su tutti i movimenti, mischiando entrate (positive) e uscite (negative). Il risultato tende a zero e non ha significato di business.
  • Effetto: KPI importo_medio praticamente inutile.
  • Fix: separare in avg_importo_entrate e avg_importo_uscite, oppure usare solo AVG(ABS(importo)) — già presente come valore_assoluto_medio.
