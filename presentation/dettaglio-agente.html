L'agente, riga per riga

Tutto l'agente vive in un solo file: backend/app/agent.py. Non è un framework custom, è un singolo LlmAgent dell'Agent Development Kit di Google. Qui sotto leggiamo il codice e lo commentiamo.

1 · Gli import: cosa serve davvero

Due classi dall'ADK (l'agente e il wrapper dei tool), più le funzioni Python che esponiamo come capacità. 

from google.adk.agents import LlmAgent
from google.adk.tools  import FunctionTool

from app.config import get_settings
from app.tools  import (
    create_github_pr,
    emit_fix_proposal,
    emit_test_plan,
    emit_test_results,
    execute_bigquery_query,
    list_available_etls,
    read_sql_code,
    save_final_report,
)
  • LlmAgent è la classe ADK che gestisce il loop ReAct: prompt → modello → tool call → tool response → loop.
  • FunctionTool trasforma una funzione Python in tool: lo schema viene generato dai type hint e dalla docstring.
  • Gli 8 tool importati sono semplici funzioni Python in app/tools.py: nessuna magia, nessuna registrazione runtime.

2 · La costante SYSTEM_PROMPT

Il prompt è una costante a livello di modulo, non viene caricata da file esterno né concatenata a runtime: è codice, vive nel diff di Git. 

SYSTEM_PROMPT = """Sei un Senior QA Engineer che testa pipeline ETL su BigQuery in modo autonomo.

Ricevi una richiesta dell'utente in italiano (una segnalazione, un dubbio, una verifica)
e la porti a termine DA SOLA usando i tool a disposizione: non esistono altri agenti.

## AMBIENTE
- Progetto BigQuery: `phrasal-method-484415-g7`.
- Repo SQL ETL + PR:  `davideconcu-accenture/Agent-GCP-ETL-Code`, branch `main`.
- Il DB è reale, le PR sono reali.

## WORKFLOW
Fase 1 — Comprensione    → list_available_etls + read_sql_code
Fase 2 — Pre-check       → INFORMATION_SCHEMA.TABLES (1 query per dataset)
Fase 3 — Pianificazione  → emit_test_plan (3-10 test)
Fase 4 — Esecuzione      → execute_bigquery_query + emit_test_results
Fase 5 — Fix & PR        → emit_fix_proposal + create_github_pr
Fase 6 — Report          → save_final_report (sempre)

## REGOLE CRITICHE  (R1..R7)
..."""     # versione abbreviata: il dettaglio è nella sotto-pagina Prompt
  • Stringa Python multilinea: niente template engine, niente runtime injection. Versionata col codice.
  • Italiano in tutto il prompt: il modello produce report e PR-body coerenti per l'utente finale.
  • Markdown nel prompt: i modelli Gemini ne sfruttano la struttura per ancorare le fasi del workflow.

3 · La factory build_agent

Una sola funzione costruisce l'agente. Il backend la chiama una volta per run e usa il valore di ritorno con il Runner ADK. 

def build_agent(model: str | None = None) -> LlmAgent:
    """Crea l'agente ADK con tutti i tool registrati."""
    s = get_settings()                    # pydantic-settings: legge env / Secret Manager

    tools = [
        FunctionTool(func=list_available_etls),    # GitHub: discovery
        FunctionTool(func=read_sql_code),           # GitHub: lettura SQL
        FunctionTool(func=execute_bigquery_query),  # BigQuery: read-only
        FunctionTool(func=emit_test_plan),          # Artefatto: piano test
        FunctionTool(func=emit_test_results),       # Artefatto: esiti test
        FunctionTool(func=emit_fix_proposal),       # Artefatto: fix proposto
        FunctionTool(func=create_github_pr),        # GitHub: apre PR
        FunctionTool(func=save_final_report),       # Artefatto: report finale
    ]

    return LlmAgent(
        name="etl_qa_agent",
        model=model or s.gemini_model,    # default: gemini-2.5-pro, override per richiesta
        description="Agente QA che testa ETL BigQuery e apre PR con i fix.",
        instruction=SYSTEM_PROMPT,            # il prompt visto sopra
        tools=tools,                          # gli 8 FunctionTool
    )

4 · Cosa dice il codice, in chiaro

Argomento model

La firma è model: str | None = None. Se la chiamata non passa nulla, l'agente cade su settings.gemini_model (default gemini-2.5-pro). Se l'utente sceglie nel frontend, il valore arriva qui — un singolo punto di override.

La lista tools

Otto FunctionTool, niente di più. L'ordine non conta: è il modello a scegliere quale chiamare, sulla base della docstring e dei type hint. Aggiungere un tool = aggiungere una riga.

name="etl_qa_agent"

Identifica l'agente nei log e negli eventi ADK. Resta stabile fra run ed è la chiave per filtrare le tracce in osservabilità.

instruction=SYSTEM_PROMPT

È il system message: ADK lo mette in cima a ogni richiesta verso Vertex AI. Il dettaglio del prompt — le 6 fasi e le 7 regole — è nella sotto-pagina dedicata.

5 · Cosa fa il backend col valore di ritorno

build_agent ritorna un LlmAgent. Il backend lo passa al Runner ADK, che gestisce il loop ReAct e produce eventi che il backend inoltra al frontend via SSE. 

agent = build_agent(model=req.model)
runner = Runner(
    app_name="vea",
    agent=agent,
    session_service=InMemorySessionService(),
)

async for event in runner.run_async(
    user_id=user_id,
    session_id=run_id,
    new_message=Content(role="user", parts=[Part(text=req.user_message)]),
):
    # event può essere: testo del modello, function_call, function_response.
    # Il backend lo normalizza e lo pubblica sul bus SSE → frontend.
    await publish_event(run_id, event)
  • Il Runner ADK è ciò che fa girare davvero il loop: noi non scriviamo prompt-loop a mano.
  • Ogni event è atomico: testo, chiamata di tool, risposta del tool. Niente buffering opaco.
  • La SessionService è in-memory: ogni run è isolato. La persistenza vive su Firestore lato backend.

6 · Perché un solo LlmAgent

  • Niente sub-agent, niente orchestratore custom: ADK fornisce già il loop ReAct robusto.
  • Un solo file da leggere: chiunque entri nel progetto vede in 60 righe come si comporta l'agente.
  • Workflow lineare: le 6 fasi del prompt + le 7 regole bastano a guidare il comportamento.
  • Tracciabilità: tutti gli eventi appartengono a un run id; debugging e audit immediati.
  • Estendibilità: aggiungere una capacità = scrivere una funzione Python e una riga in tools=[...].
